1 Ciclo de Vida de uma Atualização

Clique em cada etapa para ver o que acontece dentro do sistema quando o SAP envia uma mudança de estoque.

📡 SAP envia o webhook

O SAP detecta uma mudança de estoque no armazém e dispara imediatamente uma requisição HTTP para o middleware. Pode chegar em rajadas de até 500 por segundo.

POST http://middleware:3001/webhook/stock
Content-Type: application/json

{
  "sku_code": "SKU-CAMISETA-P",
  "product_id": "346452988",
  "variant_id": "1531149939",
  "stock": 50
}
📥 Recebido pelo Receiver (src/app.js)

🔍 Validação do payload

O Receiver verifica se todos os campos obrigatórios estão presentes. Se qualquer campo faltar, a requisição é rejeitada imediatamente — sem enfileirar nada.

// src/app.js — validação
const { sku_code, product_id, variant_id, stock } = req.body;

if (!sku_code || !product_id || !variant_id || stock == null) {
  return res.status(400).json({
    error: 'Campos obrigatórios: sku_code, product_id, variant_id, stock'
  });
}
✅ Válido → continua ❌ Inválido → HTTP 400 (SAP deve corrigir)

📦 Enfileiramento no Redis

O job é salvo imediatamente no Redis via BullMQ. O sistema responde HTTP 202 ao SAP em menos de 1 segundo — o processamento real acontece em segundo plano.

// src/queue.js — configuração do job
queue.add('update-stock', payload, {
  attempts: 5, // até 5 tentativas
  backoff: {
    type: 'exponential',
    delay: 2_000 // 2s base
  },
  removeOnFail: false // preserva na DLQ
});

// SAP recebe imediatamente:
res.status(202).json({ jobId: "42", status: "queued" });
📬 HTTP 202 → SAP confirmado ⏳ Job aguarda worker na fila

🛡️ Idempotência — filtro de duplicatas

Antes de qualquer chamada externa, o Worker verifica se esse exato par sku+estoque já foi processado nos últimos 5 minutos. Se sim, descarta silenciosamente.

// src/idempotency.js
const hash = SHA256(`${skuCode}:${stock}`);
// ex: SHA256("SKU-CAMISETA-P:50") = "a3f8..."

const existing = await redis.get(`idem:${skuCode}`);

if (existing === hash) {
  // mesmo SKU, mesmo estoque → DESCARTA
  return { skipped: true };
}

// novo estado → grava hash com TTL de 5 min
redis.setex(`idem:${skuCode}`, 300, hash);
⏭️ Duplicata → descartado (sem chamada à API) ✅ Novo estado → continua para rate limiter

🎟️ Rate Limiter — Token Bucket

Cada chamada à Nuvemshop consome um "token". O balde tem 500 tokens (= 1 s de burst) e repõe 500 por segundo. Se vazio, o worker aguarda pacientemente — sem perder o job.

Balde de tokens atual (simulação):
// src/rateLimiter.js — Token Bucket
maxTokens = 500 // capacidade do balde (= 1 s burst)
refillRate = 500/s // +500 tokens por segundo

await rateLimiter.acquire(); // aguarda se vazio
// após resposta da API:
rateLimiter.adjustCapacityFromHeader(x-rate-limit-remaining);
⏳ Sem token → aguarda (job não é perdido) ✅ Token disponível → chama API

🌐 Chamada à API Nuvemshop

O cliente HTTP faz a requisição à Nuvemshop. Se receber erros transitórios (429, 5xx), aguarda e retenta. Erros permanentes (4xx) são descartados — não há retry.

POST https://api.nuvemshop.com.br/2025-03/{store}/
      products/{productId}/variants/stock

Authentication: bearer {token} // ← NÃO é "Authorization"

{ "action": "replace", "value": 50, "id": "153..." }
200 OK → sucesso ✅ 429 / 5xx → retry ⏳ 404 / 400 → descarta ❌

🏁 Conclusão

Job concluído com sucesso — log registrado. Se todas as 5 tentativas falharem, o job vai para a DLQ para análise humana. Nenhum dado é perdido.

Retry automático & DLQ
src/nuvemshop.js · src/worker.js · BullMQ
▼

Erros transitórios são tratados com retry exponencial. Erros permanentes são descartados sem retry.

ErroComportamento
429 Rate limitRetry — espera o tempo do header reset
5xx ServidorRetry — backoff exponencial
404 Não encontradoDescarta — produto inválido na Nuvemshop
400 / 422 InválidoDescarta — corrigir na origem
401 / 403 AuthDescarta — credenciais inválidas
Rede / timeoutRetry — até esgotar tentativas
Após 5 falhasDLQ — preservado para análise humana
📊
Health & Alertas de Fila
src/app.js · GET /health/queue
▼

O endpoint /health/queue expõe o estado da fila com limiares pré-configurados. Retorna HTTP 503 em estado crítico — disparando alertas automaticamente em qualquer monitor de uptime.

🟡
Degraded — fila > 1.000 jobs ou DLQ > 10 jobs · HTTP 200
🔴
Critical — fila > 10.000 jobs ou DLQ > 100 jobs · HTTP 503
Campo na respostaSignifica
waitingJobs enfileirados aguardando worker
activeSendo processados agora
delayedEm backoff (aguardando retry)
failedNa DLQ — requer atenção
completedConcluídos na última hora

app.js

Worker — Processo 2
Motor do sistema. Consome 10 jobs simultaneamente da fila. Orquestra o ciclo completo: idempotência → rate limiter → chamada API → retry. Envia jobs falhos para DLQ após 5 tentativas.
ENTRYPOINT BullMQ Worker concurrency=10

Fila BullMQ
Configura a fila "stock-updates" no Redis. Define política de retry (5×, backoff exponencial base 2s) e limpeza de jobs concluídos (máx 1.000, janela 1h). removeOnFail: false preserva DLQ.
BullMQ Queue 5 retries

Deduplicação
Calcula SHA-256(sku_code:stock) e consulta o Redis. Se o hash já existe (TTL 5 min), o job é descartado sem chamar a API. Se novo estado, grava o hash com expiração automática.
